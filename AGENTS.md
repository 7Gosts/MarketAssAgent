# STYLE_GUIDE + 性能规则

## 使用语言
- 默认使用中文（简体）。

## 优先级：先理解再动手
1. 先定位需求在代码里的调用链、关键函数、关键参数。
2. 先判断可行性与风险点；若存在不确定性，先问清楚假设，不硬猜。
3. 确认后再修改代码。

## 代码改动原则（最少改动优先）
- 复用现有工具类/函数（例如已有 Feishu API 封装），避免引入新依赖与新架构。
- 能改一行就别重构一片，避免无关清理、风格大改、跨模块重命名。
- 删除逻辑/边界条件要与现有行为一致，必要时加兜底，避免大范围重排。
- 用户明确要求仅影响某文件时，严格限定修改范围。

## 精简与可读性
- 优先用清晰且短的代码完成目标，不引入绕路中间过程。
- 条件判断尽量用正逻辑，主路径放在 affirmative 分支；必要时用 guard clause。
- 变量与分支要让读者一次阅读可理解输入、输出与原因。
- 可由上游显式传入的分支信息，不在脚本内做隐式推断。
- 存在表头定义与行数据结构时，二者同步更新并保证列顺序一致。

## 必须交付的中间产物（协作节奏）
- 修改前后提供关键 `git diff` 片段。
- 涉及行为变更（删除/写入/鉴权/类型变化）时，说明失败模式、触发条件、验证/兜底。
- 目标文件未被 git 跟踪（`??`）时，明确提示普通 `git diff` 不显示该文件。

## 验证策略（先最小验证）
- 修改飞书/外部接口逻辑时，先跑最小脚本验证返回结构/类型，再做语法检查。
- 无法完整集成验证时，说明验证缺口与建议下一步。

## 询问策略（当规则不够时）
- 需要决定顺序/末尾/删除集合/是否排除目标等语义时，先询问确认。
- 用户已明确规则时，严格按用户规则执行，不额外发散。

## 分组扫描性能规则
- 处理表格/列表同步时，若存在同一个 key 连续多条数据，避免每条都从头扫描目标数据。
- 优先使用 `key -> scan_cursor`（或等价索引）记录下次起扫位置，并在每次写入后更新游标。
- 允许游标失效时回退到全表扫描，但先尝试“游标到末尾，再从头到游标”的两段扫描。
- 在说明里给出复杂度（原方案 vs 优化方案），并说明瓶颈是否在网络 I/O。
